Three models are available: scx-tts (the canonical voice‑cloning model — supply a short reference clip and the model speaks in that voice), and tts-1 / tts-1-hd (preset voice catalogs with six built-in accents). All three share the same wire format, so you can switch between them by changing the model field. Quick start The speech endpoint takes three key inputs: the model, the text to convert to audio, and the voice to use. Choosing a model Model Voice mechanism When to use scx-tts Default reference voice, or per‑request voice cloning. Clone a specific speaker from a 5–10 second reference clip, or fall back to a clean neutral default. Pick this when the preset voices don&#x27;t match your brand or character. tts-1 Preset voice catalog — pick by name or UUID. Latency‑tuned. Right for chat/agent UX where speed matters and any of the six preset accents fit. tts-1-hd Same preset catalog as tts-1 . Quality‑tuned. Slightly slower, marginally cleaner output for offline / podcast / narration use cases. The three models share the same request body, response stream, and error shape — you swap between them by changing only the model field. Preset voices ( tts-1 , tts-1-hd ) Pick a preset by passing its name in the voice field. These apply to tts-1 and tts-1-hd only — scx-tts ignores the voice field and uses its default reference (or your custom clone, see below). Voice Description australian-sam Male, Australian accent - natural and friendly friendly-kiwi Male, New Zealand accent - casual and approachable likeable-aussie Female, Australian/NZ accent - likeable and pleasing ito Male, American accent - warm and conversational serene-assistant Female, American accent - calm and professional alice-bennett Female, British accent - professional and articulate Voice cloning with scx-tts scx-tts can speak in a voice you supply at request time. Pass a 5–10 second reference clip in the voice_ref_wav_b64 field; the model extracts a speaker embedding from the clip and conditions the synthesis on it. No upfront enrolment, no separate voice‑management API — just send the clip with each call. There are two cloning modes, controlled by whether you also pass ref_text : Mode When Output character Embedding‑only ref_text omitted (default) Uses only the speaker timbre. Cleaner output, stylistically neutral. Good for &quot;make this voice say something&quot; use cases. Full cloning ref_text supplied alongside the clip Conditions on the reference prosody too. Reproduces cadence and intonation more faithfully, at some cost in cleanliness. Default reference voice (no cloning) If you just want to use scx-tts without supplying your own voice, omit voice_ref_wav_b64 entirely. Longer clips don&#x27;t improve quality and cost more bandwidth on every call. Match the languages : pair an English reference with English input (or French↔French, etc.). Cross‑language pairs produce unpredictable output. Clean recordings beat long ones : a 6‑second clip recorded in a quiet room with a decent mic clones better than a 30‑second clip with HVAC hum or a phone speaker. Container format : voice_ref_wav_format defaults to wav . Set it to mp3 or pcm if your clip is in a different container. Cache the encoded reference client‑side : the clip is uploaded on every call, so don&#x27;t re‑base64 it inside a hot loop. For tts-1 and tts-1-hd , split text into chunks and generate audio for each segment if you need to go beyond the per‑request limit. For scx-tts , the model auto‑splits long inputs at sentence boundaries internally and stitches the result into a single streamed response — you don&#x27;t need to chunk client‑side under the 5000‑char cap. Sentences over ~400 characters fall back to clause‑level splits so a single un‑terminated paragraph can&#x27;t blow past the per‑call audio budget. Voice cloning sends the reference clip on every call (no server‑side voice store today). Cache the base64 string client‑side if you re‑use the same voice across many requests.
