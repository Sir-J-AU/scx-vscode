Model Model Voice mechanism When to use scx-tts Clean Australian-accent default, or per-request voice cloning. Use the default voice for production TTS, or voice-clone a specific speaker from a short reference clip. The public SCX speech surface is scx-tts . Other model aliases may exist internally, but client-facing integrations should target scx-tts . Voice selection Omit voice to use the clean Australian-accent default selected at the gateway level. For reusable cloned voices, first enroll the reference clip with POST /v1/audio/voices , then pass the returned stored voice_... ID in voice . This is the recommended production path because clients upload the reference clip once and reuse the ID across HTTP speech and realtime TTS. Preset voice names such as ito or alice-bennett are not a Wordcab voice catalog. To clone from a reference clip without first enrolling a stored voice, use the voice-cloning fields below. Voice cloning Send a base64-encoded reference clip in voice_ref_wav_b64 . The model extracts a speaker embedding from the clip and conditions the synthesis on it. Mode When Output character Embedding-only ref_text omitted (default) Uses only the speaker timbre. Cleaner output, stylistically neutral. Good for &quot;make this voice say something&quot;. Full cloning ref_text supplied alongside the clip Conditions on prosody too. Reproduces cadence and intonation more faithfully, at some cost in cleanliness. Tips 5–10 seconds of clean speech is plenty. Longer clips don&#x27;t improve quality. Match the reference language to the target language. A Spanish reference paired with English input produces unpredictable output. voice_ref_wav_format defaults to wav . Set it to mp3 or pcm if your clip is in a different container. Request-time voice cloning is available on POST /v1/audio/speech . The realtime WebSocket endpoint can use stored voice_... IDs via audio.output.voice , but does not accept inline voice_ref_wav_b64 , ref_text , or x_vector_only_mode in session.update / response.create ; attempting that returns unsupported_feature . Output formats and streaming response_format Content-Type Use case wav (default) audio/wav Lossless 16-bit PCM in a WAV container. Easy to play with any decoder; ~44 bytes of header overhead. mp3 audio/mpeg Lossy, smallest payload. Downloads, podcast-style playback, anywhere file size matters. pcm audio/pcm Raw signed 16-bit little-endian PCM, mono, 24 kHz. No container framing — feed bytes directly into an AudioContext , FFmpeg pipeline, or sample-rate converter. Best for low-latency playback. All three formats are streamed — the response body&#x27;s first bytes arrive while later parts are still synthesising. Set response_format explicitly in production so clients do not depend on defaults. Speed The speed parameter is accepted by scx-tts and can shorten output at faster values. Treat it as a model hint rather than a precise linear multiplier: test the exact values you expose in production and avoid promising exact duration scaling. Long inputs input is hard-capped at 5000 characters per request — anything longer returns invalid_request_error before any synthesis happens. For scx-tts , the model has a per-call audio budget of ~39 seconds (controllable via max_new_tokens ). Long input strings are split at sentence boundaries internally and stitched into a single streamed response, so audio length scales linearly with input length — you don&#x27;t need to chunk client-side . Sentences over ~400 characters fall back to clause-level splits ( ,;: ) so a single un-terminated paragraph can&#x27;t blow past the budget. Most callers can ignore max_new_tokens . Tune it down if you want to cap synthesis latency for unbounded user inputs; tune it up only if you genuinely need single-call audio longer than ~39 s and your input has no natural sentence boundaries to chunk on. Usage & billing Each request counts as one TTS call regardless of how many internal chunks the synthesis pipeline emits. input_tokens is estimated from input character count, output_tokens from synthesised audio duration. Usage is recorded with the endpoint: /v1/audio/speech label so HTTP-mode and realtime-mode TTS traffic are grouped together in your dashboard. Voice cloning ( voice_ref_wav_b64 ) does not change the per-call price; the reference clip is processed alongside the synthesis call. The server splits long inputs at sentence boundaries internally so audio length scales with input length — you don&#x27;t need to chunk client-side. voice string Optional stored Wordcab voice ID, e.g. `voice_...`. Omit this field for the clean Australian-accent default, or use `voice_ref_wav_b64` for per-request voice cloning. Preset voice names such as `ito` are not a Wordcab voice catalog. response_format &quot;mp3&quot; | &quot;wav&quot; | &quot;pcm&quot; default: &quot;wav&quot; Audio container/codec. `wav` is the current default. `mp3` = lossy, smallest payload. `wav` = lossless 16-bit PCM in a WAV container. `pcm` = raw signed 16-bit LE PCM, mono, 24 kHz — feed directly into a low-latency playback pipeline. Response is always streamed; readers can begin playback before the body completes. mp3 wav pcm speed number default: 1 Speech speed hint between 0.25 and 4.0. Current `scx-tts` accepts this field and can shorten output at faster values, but the effect is not guaranteed to be linear across the full range; test the values you expose in production. voice_ref_wav_b64 string **`scx-tts` only.** Base64-encoded reference audio used as the voice for cloning. 5–10 seconds is recommended; longer clips don&#x27;t improve quality but cost more bandwidth. The reference is uploaded on every call — there is no server-side voice store yet, so the client is responsible for caching reference clips. voice_ref_wav_format &quot;mp3&quot; | &quot;wav&quot; | &quot;pcm&quot; **`scx-tts` only.** Container format of `voice_ref_wav_b64`. Defaults to `wav`. Use `pcm` for raw 16-bit LE 24 kHz mono. mp3 wav pcm ref_text string **`scx-tts` only.** Exact transcript of `voice_ref_wav_b64`. Required for full prosodic voice cloning — the model conditions on both the speaker embedding and the reference prosody. Omit it to fall back to embedding-only mode (cleaner output, less stylistic match). x_vector_only_mode boolean **`scx-tts` only.** When `true`, use only the speaker embedding from `voice_ref_wav_b64` and skip prosodic conditioning on `ref_text`. Defaults to `true` when `ref_text` is omitted alongside a reference clip. max_new_tokens integer **`scx-tts` only.** Cap on generated audio tokens per synthesis call. Default 512 (~39 seconds of audio). Range 30–4096. Most callers shouldn&#x27;t tune this — long inputs are auto-chunked at sentence boundaries internally so the cap only matters for ungrounded prose without sentence terminators.
