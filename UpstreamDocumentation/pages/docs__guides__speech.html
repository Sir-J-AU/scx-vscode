Speech

The Audio API provides a speech endpoint for converting text into lifelike spoken audio. Three models are available: scx-tts (the canonical voice‑cloning model — supply a short reference clip and the model speaks in that voice), and tts-1 / tts-1-hd (preset voice catalogs with six built-in accents). All three share the same wire format, so you can switch between them by changing the model field.

Quick start

The speech endpoint takes three key inputs: the model, the text to convert to audio, and the voice to use. A simple request looks like this:

from openai import OpenAI
from pathlib import Path

client = OpenAI(
    base_url="https://api.scx.ai/v1",
    api_key="your-scx-api-key",
)

response = client.audio.speech.create(
    model="tts-1",
    voice="ito",
    input="Hello! Welcome to SCX.ai text-to-speech.",
)

Path("output.mp3").write_bytes(response.content)

Choosing a model

ModelVoice mechanismWhen to use
scx-ttsDefault reference voice, or per‑request voice cloning.Clone a specific speaker from a 5–10 second reference clip, or fall back to a clean neutral default. Pick this when the preset voices don't match your brand or character.
tts-1Preset voice catalog — pick by name or UUID.Latency‑tuned. Right for chat/agent UX where speed matters and any of the six preset accents fit.
tts-1-hdSame preset catalog as tts-1.Quality‑tuned. Slightly slower, marginally cleaner output for offline / podcast / narration use cases.

The three models share the same request body, response stream, and error shape — you swap between them by changing only the model field.

Preset voices (tts-1, tts-1-hd)

Pick a preset by passing its name in the voice field. These apply to tts-1 and tts-1-hd only — scx-tts ignores the voice field and uses its default reference (or your custom clone, see below).

VoiceDescription
australian-samMale, Australian accent - natural and friendly
friendly-kiwiMale, New Zealand accent - casual and approachable
likeable-aussieFemale, Australian/NZ accent - likeable and pleasing
itoMale, American accent - warm and conversational
serene-assistantFemale, American accent - calm and professional
alice-bennettFemale, British accent - professional and articulate

Voice cloning with scx-tts

scx-tts can speak in a voice you supply at request time. Pass a 5–10 second reference clip in the voice_ref_wav_b64 field; the model extracts a speaker embedding from the clip and conditions the synthesis on it. No upfront enrolment, no separate voice‑management API — just send the clip with each call.

There are two cloning modes, controlled by whether you also pass ref_text:

ModeWhenOutput character
Embedding‑onlyref_text omitted (default)Uses only the speaker timbre. Cleaner output, stylistically neutral. Good for "make this voice say something" use cases.
Full cloningref_text supplied alongside the clipConditions on the reference prosody too. Reproduces cadence and intonation more faithfully, at some cost in cleanliness.

Default reference voice (no cloning)

If you just want to use scx-tts without supplying your own voice, omit voice_ref_wav_b64 entirely. The endpoint uses its default reference for a clean, neutral output:

from openai import OpenAI
from pathlib import Path

client = OpenAI(
    base_url="https://api.scx.ai/v1",
    api_key="your-scx-api-key",
)

response = client.audio.speech.create(
    model="scx-tts",
    input="Hello! This is the SCX text-to-speech model.",
    response_format="mp3",
)

Path("output.mp3").write_bytes(response.content)

Cloning a custom voice

Load your reference clip, base64‑encode it, and pass it as voice_ref_wav_b64. Supply ref_text (the exact transcript of the clip) for full prosodic cloning, or omit it for embedding‑only mode.

import base64
import requests
from pathlib import Path

ref_b64 = base64.b64encode(Path("reference-voice.wav").read_bytes()).decode()

response = requests.post(
    "https://api.scx.ai/v1/audio/speech",
    headers={"Authorization": "Bearer your-scx-api-key"},
    json={
        "model": "scx-tts",
        "input": "This sentence will be spoken in the voice from the reference clip.",
        "response_format": "mp3",
        "voice_ref_wav_b64": ref_b64,
        "voice_ref_wav_format": "wav",
        # Supply ref_text for full prosodic cloning; omit for embedding-only mode.
        "ref_text": "Hello, this is a sample of my speaking voice for cloning.",
    },
)
response.raise_for_status()
Path("cloned.mp3").write_bytes(response.content)

Tips for good clones

  • Clip length: 5–10 seconds of clean speech is plenty. Longer clips don't improve quality and cost more bandwidth on every call.
  • Match the languages: pair an English reference with English input (or French↔French, etc.). Cross‑language pairs produce unpredictable output.
  • Clean recordings beat long ones: a 6‑second clip recorded in a quiet room with a decent mic clones better than a 30‑second clip with HVAC hum or a phone speaker.
  • Container format: voice_ref_wav_format defaults to wav. Set it to mp3 or pcm if your clip is in a different container.
  • Cache the encoded reference client‑side: the clip is uploaded on every call, so don't re‑base64 it inside a hot loop.

Streaming raw PCM for low‑latency playback

scx-tts supports the same streaming behaviour as the preset models. Combined with response_format: pcm, you get raw 16‑bit signed LE samples at 24 kHz mono — perfect for piping straight into a playback pipeline without any container parsing.

python
import requests, sounddevice as sd, numpy as np

resp = requests.post(
    "https://api.scx.ai/v1/audio/speech",
    headers={"Authorization": "Bearer your-scx-api-key"},
    json={
        "model": "scx-tts",
        "input": "Streaming PCM straight to the speaker.",
        "response_format": "pcm",
    },
    stream=True,
)
resp.raise_for_status()

# Raw 16-bit LE PCM at 24 kHz mono.
with sd.OutputStream(samplerate=24000, channels=1, dtype="int16") as out:
    for chunk in resp.iter_content(chunk_size=4096):
        if chunk:
            out.write(np.frombuffer(chunk, dtype="<i2"))

Supported output formats

The default response format is MP3, but other formats are available:

FormatDescription
mp3Default format. Widely supported, good compression.
wavUncompressed audio. Higher quality, larger file size.
pcmRaw audio samples. Useful for further processing.

Streaming real-time audio

The Speech API supports real-time audio streaming using chunk transfer encoding. This means audio can begin playing before the full file is generated.

from openai import OpenAI

client = OpenAI(
    base_url="https://api.scx.ai/v1",
    api_key="your-scx-api-key",
)

response = client.audio.speech.create(
    model="tts-1",
    voice="serene-assistant",
    input="This is a streaming test. The audio will start playing before generation completes.",
)

# Stream to a file
with open("stream_output.mp3", "wb") as f:
    for chunk in response.iter_bytes():
        f.write(chunk)

Adjusting speed

You can adjust the speed of the generated audio by setting the speed parameter. Values range from 0.25 (slowest) to 4.0 (fastest), with 1.0 being the default normal speed.

# Slower speech (0.75x speed)
response = client.audio.speech.create(
    model="tts-1",
    voice="alice-bennett",
    input="This will be spoken more slowly.",
    speed=0.75,
)

# Faster speech (1.5x speed)
response = client.audio.speech.create(
    model="tts-1",
    voice="alice-bennett",
    input="This will be spoken more quickly.",
    speed=1.5,
)

Limitations

  • Maximum input text length is 5000 characters per request.
  • For tts-1 and tts-1-hd, split text into chunks and generate audio for each segment if you need to go beyond the per‑request limit.
  • For scx-tts, the model auto‑splits long inputs at sentence boundaries internally and stitches the result into a single streamed response — you don't need to chunk client‑side under the 5000‑char cap. Sentences over ~400 characters fall back to clause‑level splits so a single un‑terminated paragraph can't blow past the per‑call audio budget.
  • Voice cloning sends the reference clip on every call (no server‑side voice store today). Cache the base64 string client‑side if you re‑use the same voice across many requests.
